# AGENTS.md

## プロジェクト

ブラウザゲーム「VOID SURVIVORS: CHROMA ABYSS」を開発する。

* Repository: `ryomashibaba/void-survivors`
* Public URL: https://ryomashibaba.github.io/void-survivors/
* Main branch: `main`
* 構成: HTML / CSS / Canvas / Vanilla JavaScript
* セーブ: `localStorage`
* 公開: GitHub Pages

主要ファイル:

* `index.html`
* `assets/css/styles.css`
* `assets/js/game.js`
* `apply-update.bat`
* `apply-update.ps1`

npm、フレームワーク、ビルドツール、外部ゲームエンジンは、ユーザーの明示的な許可なしに追加しない。

## 作業開始前

コードを変更する前に必ず次を行う。

1. `git status`、現在のブランチ、`HEAD`の完全なコミットSHAを確認する。
2. `git fetch origin`を実行し、最新の`origin/main`を確認する。
3. ローカルが最新版でない場合や未コミット変更がある場合は、勝手に上書きしない。
4. 変更対象ファイルの最新版を全文確認する。
5. 関数、変数、DOM ID、保存キーなどの存在を検索して確認する。

過去の会話、古いファイル、以前のパッチ、コード断片だけを基準に変更しない。存在しない実装を推測しない。

## 変更方針

* 依頼に必要な範囲だけを変更する。
* 関係のない大規模リファクタリングを行わない。
* 既存機能を理由なく削除しない。
* ファイル分割は段階的に行い、一度に大規模変更しない。
* GitHub Pagesで動く相対パスを使用する。
* `localStorage`の既存キーを理由なく変更・削除しない。
* 保存形式を変える場合は旧データの移行処理を作る。
* セーブ破損や古いデータでゲーム全体が起動不能にならないようにする。

## ゲームとビジュアル

目標は、実際に公開されているインディーゲームに近い完成度へ引き上げること。

ゲーム性、操作感、ビルド構築、武器、敵、ボス、UI、演出、バランス、パフォーマンス、周回要素、セーブ安定性を総合的に改善する。

既存の鮮やかなコラージュ・ポスター・プリズム系デザインを維持する。青紫グラデーション、過剰なネオン、半透明パネルを多用した一般的なAI風UIにしない。

文字やUIを変更するときは、重なり、はみ出し、改行、戦闘視認性も確認する。

## ゲーム設計

* アップグレードは小さな数値上昇だけに偏らせない。
* 武器ごとに攻撃方法と役割を明確に変える。
* 敵を色違いだけにしない。
* ボスをHPが高いだけの敵にしない。
* 攻撃予告や危険な弾をエフェクトで隠さない。
* 数値を変える場合は旧値、新値、理由を報告する。
* 依頼と無関係なバランス値をまとめて変更しない。

## パフォーマンスとバグ

症状を推測だけで決めつけず、発生タイミングと実際のコード経路を確認する。

特に確認するもの:

* 毎フレームの生成・描画・DOM更新
* 大量の`shift`や`splice`
* パーティクル、弾、音声の大量生成
* 敵撃破時の連鎖処理
* 重複ダメージや重複死亡処理
* イベントリスナーやゲームループの多重登録
* 一時停止中の時間進行
* 再開時の入力や状態残り
* FPS依存の処理

特定イベントだけが重い場合、原因確認前に全体の画質、敵数、弾数を下げない。

## 検証

変更後は可能な範囲で次を行う。

* JavaScript構文チェック
* 変更差分の確認
* 参照切れ、重複宣言、相対パスの確認
* セーブ互換性の確認
* 利用可能ならブラウザで起動・操作・コンソールエラーを確認

実施していないテストを「確認済み」と報告しない。

## Git・パッチ

Unified Diffを出す場合:

1. 最新の完全な変更前ファイルを取得する。
2. 完全な変更後ファイルを生成する。
3. `git diff`で差分を機械生成する。
4. 同じ最新版に対して`git apply --check`を実行する。
5. 成功した同一のパッチだけを提示する。

ハンク行数を手作業で作らない。パッチ内で`...`による省略をしない。

GitHubへ直接更新する場合、実際に書き込み権限がある場合だけ実行する。Commit・Push後は新しいコミットSHAとGitHub上の保存内容を確認する。

## 最終報告

必ず次を区別して報告する。

* 実際に変更したファイル
* 開始時と更新後のコミットSHA
* 実行した構文チェック
* 実行した差分チェック
* 実行したブラウザテスト
* 未実施のテスト
* ユーザー側で確認が必要な点

実際に成功していない操作を、反映済み、Push済み、確認済みと表現しない。
